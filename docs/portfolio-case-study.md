# PokéSim KR 포트폴리오 사례

이 문서는 PokéSim KR을 PM·프로덕트 엔지니어·프론트엔드·풀스택 포트폴리오에서
설명하기 위한 원본 사례집이다. 이력서에는 여기서 2~3개 성과만 골라 쓰고,
면접에서는 문제와 판단 근거, 트레이드오프, 결과 순으로 설명한다.

## 1. 한눈에 보기

> 한국 포켓몬 TCG 사용자가 실제 구매 단위인 한 박스를 시뮬레이션하고,
> 결과의 운을 비교하며 힛카드를 수집할 수 있는 비영리 팬 서비스입니다.
> 핵심 시뮬레이션은 로그인 없이 제공하고, 기록 보관이 필요한 사용자만
> Google 계정을 선택하도록 설계했습니다.

| 항목 | 내용 |
| --- | --- |
| 역할 | 1인 기획·디자인·프론트엔드·데이터·DB·배포·운영 |
| 제품 | 박스깡·자판기깡, 누적 운, 힛카드 기록, 선택형 Google 기록 보관 |
| 기술 | Next.js 15, React 19, TypeScript, Tailwind CSS 4, Supabase, Cloudflare Workers·R2 |
| 데이터 | 한국 공식 카드 DB를 기준으로 일본판 누락 데이터와 시세 참고값 보강 |
| 운영 원칙 | 핵심 경험은 익명, 정적 데이터 우선, 이미지 CDN 분리, 쓰기는 작고 제한적으로 |
| 검증 기준 | 빌드·lint, 세트 데이터·봉입률·기록 도달성·기록 병합 검증, 운영 스모크 테스트 |

## 2. 문제 정의

한국 사용자가 기존 해외 카드 시뮬레이터를 사용할 때는 다음 문제가 있었다.

1. 한국판 카드명과 발매 세트가 맞지 않았다.
2. 실제 구매 경험은 박스 단위인데 낱팩 중심 서비스가 많았다.
3. 한국판 공식 봉입률이 공개되지 않아 결과가 그럴듯해 보여도 근거를 알기 어려웠다.
4. 한 번 즐기고 브라우저를 닫으면 기록이 사라져 재방문 동기가 약했다.
5. 카드 이미지가 많아질수록 원본 이미지를 직접 제공하는 방식은 성능과 비용이 커졌다.

제품 목표는 **실제 카드깡의 재미를 가볍게 재현하면서도, 추정 데이터의 한계를 숨기지
않고, 1인 운영 비용 안에서 세트와 사용자가 늘어날 수 있게 만드는 것**이었다.

익명 핵심 경험의 사용량과 재방문·기록 보관 전환을 함께 본다.

## 4. 핵심 제품 결정

### 결정 A. 실제 사용 단위인 박스깡을 중심에 둔다

- **가설:** 한국 사용자는 낱팩보다 한 상자 결과와 봉입 구성을 비교하는 데 더 큰 재미를 느낀다.
- **구현:** 박스 보장 슬롯을 먼저 만들고 팩 위치를 섞어, 한 박스 전체 구성을 제어한다.
- **보완:** 짧게 즐기려는 사용자를 위해 1~10팩 자판기깡을 별도 진입점으로 둔다.
- **측정:** 박스 시작률, 완주율, 다시 개봉하기, 운 확인 이동률을 구분해 수집한다.

### 결정 B. 공식 비공개 확률을 정확한 숫자인 것처럼 말하지 않는다

- 세트별 일본판 실측과 커뮤니티 합의를 출처·표본과 함께 기록한다.
- 박스 보장 슬롯과 1팩 확률을 분리한다.
- 근거가 확인된 특수팩만 모델링하고 공통 “갓팩 확률”은 만들지 않는다.
- 새 세트는 카드 데이터만 추가하지 않고 시뮬·운 분포 검증을 함께 통과시킨다.

이 결정은 단기적으로 숫자를 단정하는 화면보다 덜 화려하지만, 서비스 신뢰와 유지보수성을
높인다. 틀린 모델을 고쳤을 때도 왜 바뀌었는지 추적할 수 있다.

### 결정 C. 로그인은 핵심 기능의 입장권이 아니라 기록 보관 옵션이다

- 비로그인 사용자도 박스깡, 자판기깡, 누적 운, 로컬 힛카드 기록을 모두 사용한다.
- 사용자가 기록 보관을 선택할 때만 Google 로그인을 시작한다.
- 로그인 전 기록은 자동으로 빼앗지 않고 계정에 합칠지 직접 묻는다.
- 서버 저장 성공 전에는 게스트 원본을 삭제하지 않는다.
- 로그아웃 시 계정 기록을 게스트 영역으로 복제하지 않아 소유 경계를 유지한다.

**제품 관점:** 회원 수를 늘리기 위해 기록을 잠그는 대신, 사용자가 기록의 가치를 느낀
시점에 로그인 이유를 제시했다.

**개발 관점:** local-first 구조에 선택형 cloud sync를 붙여 인증 장애가 나도 시뮬레이션과
로컬 기록은 계속 사용할 수 있게 했다.

### 결정 D. 사용자 가치가 낮은 서버 비용은 만들지 않는다

- 시뮬레이션과 운 계산은 브라우저에서 실행해 서버 연산 비용을 없앴다.
- 카드 메타는 정적 JSON으로 배포한다.
- 이미지 트래픽은 Cloudflare R2와 256·512 WebP variant로 분리한다.
- Supabase는 인증, 제한된 기록 백업, 익명 분석 집계에만 사용한다.
- 사용자 기록은 개봉별 행이 아니라 사용자당 최대 64KB 스냅샷 한 행으로 제한한다.

## 5. 대표 사례

### 사례 1. 세트별 봉입률 시뮬레이터

**문제:** 일반 확장팩, 하이클래스팩, MEGA, 특수팩은 박스 구성과 고레어 슬롯 규칙이
다르다. 범용 확률표 하나로 처리하면 특정 카드가 나오지 않거나, 같은 고레어가
비정상적으로 반복되거나, 고정 슬롯을 “운이 좋다”고 평가하는 문제가 생겼다.

**접근**

- UI가 사용하는 공개 API와 세트별 시뮬레이션 구현을 분리했다.
- rarity·card_type별 카드 풀, 박스 슬롯, 팩 조립, 중복 방지를 각각 모듈화했다.
- seedrandom으로 같은 시드를 재현할 수 있게 했다.
- 운 계산은 결과를 재추첨하지 않고 저장된 rarity 요약을 기대 분포와 비교한다.
- 고정 슬롯은 운 점수 baseline에서 차감하고 희귀 변동 슬롯에 더 큰 가중치를 둔다.

**결과**

- 77개 세트를 같은 공개 API로 제공하면서 세트별 예외를 내부 모델에 격리했다.
- 누락 카드와 도달 불가능한 힛카드를 대량 시뮬레이션으로 검사할 수 있게 됐다.
- 새로운 rarity가 기존 문자열에 흡수되는 문제를 검증 절차와 SSOT 결정으로 남겼다.

**근거:** [시뮬레이터 구조](simulator-architecture.md),
[공개 API](../frontend/lib/simulator.ts), [세트별 모델](../frontend/lib/simulation/model.ts),
[운 계산](../frontend/lib/luck.ts)

### 사례 2. 힛카드 기록과 선택형 계정 보관

**문제:** 로컬 기록은 재방문 가치를 만들지만 다른 기기에서 이어지지 않았다. 반대로
첫 화면부터 가입을 강제하면 가벼운 시뮬레이션의 장점과 전환율을 해칠 수 있었다.

**접근**

- 게스트 기록과 사용자별 로컬 캐시를 다른 키로 분리했다.
- 서버에는 일반 카드와 이미지·가격을 저장하지 않고 압축된 세트 합계와 힛카드 키만 저장했다.
- 사용자당 한 행, 64KB, 최대 8개 기기 source라는 명시적 예산을 뒀다.
- 15초 동안 변경을 모아 저장하고 revision 낙관적 잠금으로 기기 충돌을 감지했다.
- RLS와 auth.uid() 기반 RPC로 본인 기록만 읽고 쓸 수 있게 했다.
- 계정 삭제 시 Auth 사용자와 백업을 cascade로 삭제하되 게스트 기록은 유지했다.

**결과**

- 인증이 없어도 기존 경험이 온전히 동작한다.
- 로그인 사용자는 다른 브라우저에서 기록을 복원할 수 있다.
- DB 행 증가량이 개봉 수가 아니라 사용자 수에 비례한다.
- 병합, 중복 방지, 소스 제한, 계정 삭제 경계를 자동 검증한다.

**근거:** [기록 보관 설계](record-backup.md),
[백업 모델](../frontend/lib/recordBackup.ts), [동기화 흐름](../frontend/lib/useRecordBackup.ts),
[DB 마이그레이션](../supabase/migrations/20260713000010_user_record_backup.sql)

### 사례 3. Supabase Free Plan 비용 최적화

**문제:** 익명 이벤트 약 197만 건이 쌓이며 DB가 953MB로 증가해 Free Plan 500MB
한도를 넘었다. 원본을 모두 지우면 DAU, 리텐션, 누적 박스·팩 통계를 잃게 된다.

**접근**

- pg_stat_user_indexes에서 크기와 실제 스캔 횟수를 함께 조사했다.
- 사용되지 않는 대형 B-tree 인덱스 7개를 제거했다.
- append-only 시간 범위 조회는 43MB B-tree 대신 24KB BRIN을 사용했다.
- 원본은 sim 14일, user 30일만 유지하고 장기 지표는 일별 rollup으로 분리했다.
- pg_cron으로 집계 후 정리를 자동화했다.
- 전체 이벤트를 매번 읽어 500이 나던 글로벌 통계 RPC는 쓰기 시 누적하는 singleton
  캐시로 교체했다.

**결과**

- DB를 953MB에서 482MB로 약 49.4% 줄였다.
- DAU, D1·D7 리텐션, 일별 시뮬, 전체 누적값은 계속 보존했다.
- 글로벌 통계 조회를 한 행 조회로 바꿔 간헐적 statement timeout을 없앴다.
- 즉시 유료 플랜으로 전환하지 않고 운영 여유를 확보했다.

**근거:** [비용 최적화](supabase-analytics-cost-optimization.md),
[DB 진단](../supabase/queries/database-size-audit.sql),
[통계 캐시](../supabase/migrations/20260715000013_cache_global_stats.sql)

### 사례 4. 반복 가능한 카드 세트 데이터 파이프라인

**문제:** 한국 공식 DB에 일부 고레어 카드 상세가 없고, 일본판 보강 데이터에는 어두운
이미지, 깨진 이름, 잘못된 rarity, 시세 누락이 섞일 수 있었다. 세트가 늘수록 수동 확인만으로
누락을 막기 어려웠다.

**접근**

- data/sets를 원본 SSOT로, frontend/public/sets를 배포 사본으로 분리했다.
- 한국 공식 카드 DB를 우선하고 누락 카드는 일본판 목록과 이미지로 보강했다.
- 카드 번호 coverage, 이미지 중복·밝기·해상도, 알려진 rarity, 힛카드 도달성을 검사했다.
- 일본 참고가에 환율과 명시적 보정계수를 적용하고 출처를 데이터에 남겼다.
- 이미지는 256·512 WebP로 변환해 R2에 업로드했다.

**결과**

- 77세트 10,884개 카드 레코드를 같은 스키마와 검증 절차로 관리한다.
- 수집 → 보강 → 검증 → 운 분포 → 이미지 최적화 → 배포를 체크리스트로 표준화했다.
- 카드 데이터와 시뮬 로직을 같은 작업 단위로 묶어 “보이지만 나오지 않는 카드”를 줄였다.

**근거:** [세트 파이프라인](card-set-pipeline.md),
[이미지 전송 구조](image-delivery.md), [데이터 검증](../scripts/validate-card-data.ts),
[기록 도달성 검사](../scripts/audit-hit-dex-coverage.ts)

## 6. 포지션별 강조점

### PM 지원 시

1. 해외·낱팩 중심 서비스와 한국 사용자 구매 경험의 차이를 문제로 정의했다.
2. 한국판 데이터, 박스 단위 결과, 운 비교를 첫 사용 가치로 우선했다.
3. 가입 강제 대신 기록 가치가 생긴 뒤 선택형 로그인을 제안했다.
4. 회원 수만 보지 않고 익명 사용량, 재방문, 기록 보관 전환을 함께 보도록 설계했다.
5. 거래·광고·커뮤니티를 비목표로 두고 시뮬 품질과 데이터 신뢰에 집중했다.
6. 지표를 잃지 않으면서 데이터 보존 수준을 계층화해 무료 한도 안으로 복귀했다.

PM 면접에서는 구현보다 **어떤 사용자 문제를 왜 이 순서로 풀었고, 무엇을 일부러 하지
않았는지**를 먼저 말한다.

### 개발자 지원 시

1. 세트별 규칙을 모듈화한 seed 기반 클라이언트 시뮬레이션.
2. 정적 카드 데이터와 R2 이미지 CDN으로 읽기 트래픽을 서버에서 분리한 구조.
3. local-first 상태와 선택형 cloud sync의 병합·충돌·실패 복구 설계.
4. Supabase RLS, SECURITY DEFINER RPC, 최소 권한, 계정 cascade 삭제.
5. 197만 이벤트의 인덱스 사용량을 측정하고 rollup·retention·BRIN으로 최적화한 경험.
6. 전체 스캔 RPC를 쓰기 시 누적 캐시로 전환한 운영 장애 대응.

개발 면접에서는 **제약 → 대안 → 선택 → 실패 모드 → 검증** 순으로 설명한다.

## 7. STAR 면접 소재

### DB 한도 초과

- **Situation:** 이벤트가 약 197만 건으로 증가해 500MB Free Plan에서 DB가 953MB가 됐다.
- **Task:** 비용을 올리지 않으면서 장기 DAU와 리텐션을 보존해야 했다.
- **Action:** 인덱스 사용량을 조사하고 7개를 제거한 뒤 BRIN, 일별 rollup, retention,
  Cron을 설계했다.
- **Result:** 482MB로 줄이고 장기 지표는 유지했다. 읽기 전체 스캔도 singleton 캐시로 제거했다.

### 가입 강제 없이 기록 동기화

- **Situation:** 로컬 기록은 재방문 가치를 만들지만 다른 기기에서 이어지지 않았다.
- **Task:** 핵심 UX와 낮은 운영비를 유지하며 계정 보관을 제공해야 했다.
- **Action:** 게스트와 계정 로컬 영역을 분리하고 사용자당 64KB 한 행·revision·source_id
  기반 동기화를 구현했다.
- **Result:** 비로그인 경험을 보존하면서 사용자가 원할 때만 다른 기기 복원을 제공했다.

### 카드 누락과 봉입 모델 신뢰

- **Situation:** 특정 SAR가 나오지 않거나 일부 특수팩에서 비정상 중복이 발생했다.
- **Task:** 70개가 넘는 세트에서 같은 유형의 누락을 낮은 비용으로 찾아야 했다.
- **Action:** rarity 풀과 고레어 슬롯을 분리하고 coverage·1,000박스 도달성·특수팩 전용
  카드 검증을 스크립트로 만들었다.
- **Result:** 개별 제보를 일회성 수정으로 끝내지 않고 전체 세트 회귀 검사로 전환했다.

## 8. 이력서 문장 예시

### PM·프로덕트

- 한국 포켓몬 TCG 사용자의 실제 구매 단위와 기록 니즈를 바탕으로 박스깡 중심
  시뮬레이터를 기획·운영하고, 기록 가치가 생긴 시점의 선택형 Google 보관 흐름을 설계했습니다.
- 77개 세트의 데이터 수집·봉입률·이미지·누락 검증 파이프라인을 표준화하고,
  비공개 확률을 출처와 추정치로 구분하는 데이터 신뢰 원칙을 수립했습니다.
- 약 197만 건의 운영 이벤트를 분석해 핵심 지표 보존 수준을 재설계하고,
  DB 사용량을 953MB에서 482MB로 49.4% 절감했습니다.

### 프론트엔드·풀스택

- Next.js·TypeScript로 seed 재현이 가능한 세트별 박스 시뮬레이션과 운 분포 모델을
  클라이언트에 구현해 서버 연산 없이 77개 세트를 제공했습니다.
- localStorage 기반 게스트 기록에 Supabase RLS·RPC·낙관적 잠금을 결합해,
  사용자당 64KB 상한의 선택형 다중 기기 기록 동기화를 구현했습니다.
- PostgreSQL 인덱스 분석, BRIN 전환, 일별 rollup·retention·pg_cron,
  singleton 통계 캐시로 비용과 조회 안정성을 개선했습니다.
- Cloudflare R2에 256·512 WebP variant를 배포해 카드 수 증가에 따른 이미지 비용을 제어했습니다.

## 9. 3분 데모 순서

1. 박스깡과 자판기깡이 서로 다른 사용 상황을 해결한다는 점을 설명한다.
2. 박스를 열고 SAR·MUR 같은 힛카드 연출을 보여준다.
3. 누적 운에서 고정 슬롯과 희귀 변동 슬롯을 다르게 평가한다고 설명한다.
4. 힛카드 기록에서 등록·미등록 상태와 세트별 진행률을 보여준다.
5. 비로그인 상태에서도 모두 동작한 뒤 Google 보관으로 다른 브라우저에서 복원한다.
6. 아키텍처 그림과 DB 953MB → 482MB 운영 사례로 마무리한다.

## 10. 과장하지 않을 표현

- “공식 봉입률”이 아니라 **출처가 있는 추정 모델**이라고 말한다.
- 시세에는 **일본 참고가 기반 한국판 추정값**이 포함된다고 말한다.
- 회원 수와 방문 수를 혼용하지 않는다.
- DB 감소가 사용자 증가를 만들었다고 인과관계를 주장하지 않는다.
- 포켓몬 IP의 소유권을 주장하지 않고 비영리 팬 프로젝트임을 명시한다.
- 자동 테스트가 없는 UI 영역은 “운영 스모크 테스트로 확인”이라고 구분한다.

## 11. 다음에 측정할 제품 지표

| 질문 | 지표 |
| --- | --- |
| 첫 재미를 느끼는가 | 메인 → 세트 선택 → 첫 개봉 완료율 |
| 결과가 다음 행동을 만드는가 | 결과 → 다시 개봉, 운 확인, 기록 이동률 |
| 기록이 재방문을 만드는가 | D1·D7 리텐션, 기록 등록 사용자와 미등록 사용자의 차이 |
| 로그인 제안 시점이 자연스러운가 | 보관 버튼 노출 → Google 시작 → 병합 완료율 |
| 계정 보관이 실제 가치를 주는가 | 다른 기기 복원 사용자 수, 동기화 실패율 |
| 세트 운영이 효율적인가 | 세트 추가 소요시간, 누락·이미지 감사 실패 수 |

기록을 잠가 로그인 전환을 인위적으로 높이는 실험은 핵심 익명 원칙과 충돌하므로 하지 않는다.
