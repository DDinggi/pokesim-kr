# 릴리스 전 점검표

큰 UI·기록·시뮬 변경을 합치기 전에 아래 순서로 확인한다. 테스트용 도감은 별도
`localStorage` 키를 사용하므로 실제 도감 기록을 덮어쓰지 않는다.

## 1. 자동 검증

저장소 루트에서 실행한다.

```powershell
pnpm --dir frontend lint
pnpm --dir frontend build
pnpm --dir scripts validate:data -- --strict
pnpm --dir scripts validate:records
pnpm --dir scripts audit:hit-dex -- --trials=1000
```

- `lint`, `build`: TypeScript·React·Next.js 오류와 프로덕션 빌드를 확인한다.
- `validate:data`: 세트 JSON, 레어도, 이미지 키와 원본/프론트 사본 정합성을 확인한다.
- `validate:records`: 비로그인·계정별 기록 분리, 병합 멱등성, 기기 중복 제거와 저장 한도 실패를 확인한다.
- `audit:hit-dex`: 활성 세트마다 1,000박스를 열어 도감 대상 카드가 실제 시뮬에서
  전부 등장 가능한지 확인한다.
- 시뮬 확률이나 슬롯을 수정했다면 해당 세트마다 아래 명령도 실행한다.

```powershell
pnpm --dir scripts validate:luck -- --set <세트 코드> --strict
```

## 2. 로컬 실행

```powershell
pnpm --dir frontend dev
```

기본 주소는 `http://localhost:3000`이다. 이미 사용 중이면 Next.js가 안내한 다음
포트를 사용한다. 브라우저 개발자 도구의 Console과 Network 탭을 열어 둔다.

## 3. 화면 점검

다음 뷰포트를 한 번씩 확인한다.

| 구분 | 크기 |
|---|---:|
| 모바일 | 390 × 844 |
| 태블릿 | 768 × 1024 |
| 데스크톱 | 1440 × 900 |

### 메인·기록

- 누적 운과 힛카드 도감이 서로 다른 버튼으로 열린다.
- 새로고침 직후 기록 수치가 깜빡이거나 hydration 경고가 뜨지 않는다.
- 박스 하나와 자판기 팩 하나를 연 뒤 누적 운과 도감에 각각 반영된다.
- 로그아웃해도 익명 개봉 기록과 로컬 도감은 유지된다.

### 박스깡

- `MEGA`, `스칼렛&바이올렛`, `소드&실드`, `썬&문` 탭의 합계가 활성 세트 수와 같다.
- 각 탭에서 세트 하나씩 선택해 박스 결과까지 진행한다.
- 검색 결과, 탭 전환, 메인 복귀가 모바일에서도 가로로 넘치지 않는다.

### 자판기깡

- 시대 탭이 자판기 위에 있고, 선택 가능한 팩 이름은 자판기 아래에 보인다.
- 다른 탭으로 이동해도 장바구니 수량은 유지된다.
- 서로 다른 시대의 팩을 담아 개봉하고 결과·누적 운 반영을 확인한다.
- 스타트덱은 자판기 목록에서 제외된다.

### 힛카드 도감

전체 도감 레이아웃을 볼 때는 먼저 아래 명령을 실행한다.

```powershell
pnpm --dir scripts debug:fill-hit-dex
```

그다음 현재 개발 서버에서 `/debug/fill-hit-dex.html`을 연다. 예를 들어 서버가
3001 포트라면 `http://localhost:3001/debug/fill-hit-dex.html`이다.

- 미등록 카드는 `?`, 등록 카드는 카드 이미지로 보인다.
- 시대별 진행률과 확장팩 구분이 맞는다.
- 세트별 상한 안에서 기준가 7만 원 이상인 최고 힛카드에만 잔잔한 빛 효과가 보인다.
- 효과가 카드 텍스트나 클릭을 가리지 않고 스크롤이 끊기지 않는다.
- `prefers-reduced-motion`을 켜면 움직임이 멈춘다.

## 4. Google 백업

운영 Supabase 마이그레이션과 Google Provider를 적용한 환경에서 확인한다.

1. 로그아웃 상태에서 `/privacy`, `/terms`와 메인 하단 링크를 확인한다.
2. 비로그인 상태에서 카드 한 장을 도감에 등록한다.
3. `Google 계정으로 계속하기`를 누르고 로컬 기록 병합을 선택한다.
4. 새 시크릿 창 또는 다른 브라우저에서 같은 계정으로 로그인한다.
5. 병합한 카드와 획득 횟수가 복원되는지 확인한다.
6. 로그아웃한 뒤 익명 로컬 도감으로 돌아오는지 확인한다.
7. 다시 로그인했을 때 서버 도감이 그대로 복원되는지 확인한다.
8. 테스트 계정을 탈퇴해 계정 기록은 삭제되고 게스트 기록은 유지되는지 확인한다.
9. 로그인 상태에서 메인 진입과 박스·팩 개봉 후 Network 탭의 `user_events`,
   `sim_events` POST가 2xx이고 401·403 응답이 없는지 확인한다.

백업 RPC가 없다는 오류가 나오면 프론트 문제가 아니라 해당 환경에 최신
`supabase/migrations/*user_record_backup.sql`을 아직 적용하지 않은 상태다.

## 5. 합치기 전 확인

```powershell
git diff --check
git status --short
```

의도하지 않은 세트 JSON, 이미지, 마이그레이션 변경이 섞이지 않았는지 확인한 뒤
기능 단위로 커밋한다. Supabase 마이그레이션은 기존 데이터에 대한 롤백·보존 방식을
확인하고 운영에 적용한다.
